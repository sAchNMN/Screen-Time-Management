# 屏幕时间管理软件 — 开发路线

## 1. 项目概述

开发一个 Windows 桌面应用，用户可指定要监控的软件列表，系统后台实时检测这些软件的运行状态，并累计记录它们的运行时长。

---

## 2. 技术栈选择

| 层次 | 技术选型 |
|------|----------|
| 语言 | Java 17+ |
| GUI | JavaFX + SceneBuilder |
| 构建工具 | Maven 或 Gradle |
| 数据库 | SQLite（嵌入式，轻量，无需安装数据库服务） |
| 进程检测 | `ProcessHandle` API (Java 9+) / WMI 命令行 |
| 打包 | jpackage 打包为 exe 安装包 |

---

## 3. 核心功能模块

### 3.1 用户选择监控的软件
- 显示当前系统正在运行的进程列表
- 用户勾选需要监控的软件（如 Chrome、WeChat、IDE 等）
- 可手动添加不在运行列表中的可执行文件路径（.exe）
- 保存到「监控列表」

### 3.2 后台进程检测
- 定时轮询（如每 5~10 秒）系统中正在运行的进程
- 与「监控列表」比对，判断目标软件是否在运行
- 如果目标进程在运行中，则累计计时
- 如果目标进程退出，停止计时，保存本次会话时长
- 后台进程检测优先保证电脑性能，避免对系统资源造成过大压力

### 3.3 时长记录与存储
- 数据库表结构（SQLite）：
  - `monitored_apps` — 被监控的软件列表（名称、可执行文件路径）
  - `usage_records` — 使用记录（软件ID、开始时间、结束时间、持续秒数）
  - `daily_summary` — 每日汇总（软件ID、日期、总时长）
- 每次检测到进程退出时写入一条记录
- 每天凌晨汇总前一天的记录到 `daily_summary`

### 3.4 数据展示
- 今日各软件使用时长（实时刷新）
- 历史日 / 周 / 月趋势图表（柱状图或折线图）
- 软件使用排行

### 3.5 其他辅助功能
- 系统托盘常驻，后台静默运行
- 开机自启选项（开关功能）
- 数据导出（CSV / JSON）
- 可设置每日使用时间，超时提醒，提醒用户休息，用户休息完之后继续使用。
- 可自定义设置使用时间间隔，如 1 小时、2 小时等。

## 4. 开发路线（分阶段）

### 第一阶段：基础框架与 UI（1~2 周）
1. 初始化 Maven 项目，引入 JavaFX 依赖
2. 设计主窗口布局（SceneBuilder）：左侧导航 + 右侧内容区
3. 实现「监控软件管理」页面
   - 获取系统进程列表并展示
   - 勾选添加 / 移除监控目标
4. 集成 SQLite（使用 `sqlite-jdbc` 驱动）
5. 实现监控列表的 CRUD（增删改查）

### 第二阶段：后台检测引擎（1~2 周）
1. 实现进程检测核心逻辑
   - 通过 `ProcessHandle.allProcesses()` 获取系统进程
   - 根据进程名（如 `chrome.exe`）匹配监控列表
2. 设计后台定时任务（`ScheduledExecutorService`），每 5 秒轮询一次
3. 实现状态管理：检测到进程启动 → 开始计时；进程消失 → 停止计时并落库
4. 进程检测组件独立为后台 Service 层

### 第三阶段：数据记录与展示（1 周）
1. 实现 `usage_records` 表的写入与查询
2. 实现 `daily_summary` 表的自动汇总
3. 开发「今日概览」页面，展示今日各软件使用时长
4. 集成图表库（如 `JFreeChart` 或 JavaFX 原生图表）展示历史趋势

### 第四阶段：系统集成与优化（1 周）
1. 系统托盘功能（`java.awt.TrayIcon`）
2. 开机自启（注册表写入）
3. 数据导出功能
4. 使用时间超时提醒功能（用户可设置每日使用时间，超过后提醒用户休息，用户休息完后继续使用）
5. 性能优化：减少轮询开销

### 第五阶段：打包与发布（3~5 天）
1. 使用 `jpackage` 将应用打包为 exe 安装包
2. 附带 JRE（生成不含依赖问题的独立安装包）
3. 测试在不同 Windows 版本上的兼容性

## 6. 关键技术细节

### 进程检测方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| `ProcessHandle.allProcesses()` | 纯 Java API，跨平台，无需外部依赖 | 需要 Java 9+，获取的信息有限 |
| `tasklist` 命令行 | 信息详细（PID、内存等） | 每次执行创建子进程，性能开销大 |
| WMI 查询 (`wmic`) | 信息最全面 | 查询速度慢，依赖 WMI 服务 |

**推荐**：以 `ProcessHandle` 为主方案，仅在需要额外信息时使用 `tasklist`。

### 轮询间隔
- 默认 5 秒检测一次
- 检测到进程存在后，记录开始时间戳
- 检测到进程消失后，计算持续时长并写入数据库
- 短时间（< 30 秒）的运行可忽略（过滤误触）

### 数据库初始化 SQL（schema.sql）

```sql
CREATE TABLE IF NOT EXISTS monitored_apps (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    app_name    TEXT NOT NULL,
    process_name TEXT NOT NULL,   -- 如 chrome.exe
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS usage_records (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    app_id      INTEGER NOT NULL,
    start_time  TIMESTAMP NOT NULL,
    end_time    TIMESTAMP,
    duration_seconds INTEGER,
    FOREIGN KEY (app_id) REFERENCES monitored_apps(id)
);

CREATE TABLE IF NOT EXISTS daily_summary (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    app_id      INTEGER NOT NULL,
    date        DATE NOT NULL,
    total_seconds INTEGER DEFAULT 0,
    FOREIGN KEY (app_id) REFERENCES monitored_apps(id),
    UNIQUE(app_id, date)
);
```

## 7. 注意事项

- **权限**：普通用户权限即可枚举进程，无需管理员权限
- **隐私**：本软件只记录用户主动选择的软件，不偷扫所有进程，数据全部本地存储
- **资源占用**：后台检测仅做进程名匹配，内存占用控制在 50MB 以内
- **异常处理**：进程检测模块需做容错，单个检测失败不应影响整体运行
- **时区**：涉及日期汇总时注意使用系统时区
